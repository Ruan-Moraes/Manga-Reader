## 1. Visão Geral

### Objetivo

Evoluir a aplicação Manga Reader para suportar **conteúdo de domínio** em três idiomas — português brasileiro (`pt-BR`), inglês americano (`en-US`) e espanhol (`es-ES`) — preservando o idioma de exibição configurado pelo usuário em toda a aplicação, com fallback confiável e estrutura preparada para extensão a novos idiomas sem retrabalho estrutural.

Esta entrega **não é** sobre i18n de mensagens/UI (já existente via Spring `MessageSource` e `react-i18next`). O foco é o conteúdo persistido — títulos de mangá, sinopses, notícias, eventos, descrições administrativas, etc.

### Idiomas Suportados

| Código | Idioma | Papel |
|---|---|---|
| `pt-BR` | Português (Brasil) | **Default e fallback obrigatório** |
| `en-US` | Inglês (EUA) | Suportado |
| `es-ES` | Espanhol (Espanha) | Suportado |

### Regra de Fallback

Conteúdo de **catálogo/administrativo** (Title, NewsItem, Event, Tag, Group, Store, SubscriptionPlan):
- Tenta o locale do request (`Accept-Language`).
- Se ausente, cai para `pt-BR`.
- Se ainda ausente, retorna a primeira tradução disponível.
- Se vazio, retorna string vazia.

Conteúdo **gerado por usuário** (Comment, MangaRating, ForumTopic, ForumReply):
- **Não tem fallback.** É particionado estritamente por idioma.
- Cada post é gravado com o idioma em que o usuário estava configurado no momento da criação.
- Listagens públicas filtram `WHERE language = :userLocale` para evitar contaminação cross-language.

## 2. Arquitetura Funcional

### Como o conteúdo multilíngue é organizado

**Catálogo/Admin — Embedded `LocalizedString`:**
- Domain Value Object `LocalizedString` (e variantes `LocalizedText`, `LocalizedStringList`) encapsula `Map<String, String>` com chave = language tag (`pt-BR`, `en-US`, `es-ES`).
- Persistido como **JSONB** em PostgreSQL (via Hibernate `@JdbcTypeCode(SqlTypes.JSON)`).
- Persistido como **subdocumento embutido** em MongoDB.
- Adicionar um novo idioma = inserir nova chave no JSONB. Zero migração de schema.

**UGC — Particionado por idioma:**
- Cada entidade UGC ganha um campo `language` (VARCHAR(10) NOT NULL).
- Preenchido pelo backend a partir do `Accept-Language` no momento da criação. Frontend não envia esse campo.
- Listagens filtram pelo idioma do request — usuário em EN só vê posts em EN.

### Como o idioma é resolvido no sistema

**Backend (Spring Boot):**
1. Frontend envia `Accept-Language: <tag>` em todo request HTTP.
2. `AcceptHeaderLocaleResolver` (já configurado em `I18nConfig`) popula `LocaleContextHolder.getLocale()`.
3. Mappers MapStruct (camada presentation) chamam `LocaleResolutionService.resolve(LocalizedString)` ao montar DTOs públicos — recebem locale do contexto e aplicam a regra de fallback.
4. DTOs admin (`*AdminResponse`) retornam todas as traduções como `Map<String, String>` para edição.
5. Use cases UGC de criação leem `LocaleContextHolder.getLocale().toLanguageTag()` e gravam em `entity.language`. Use cases de listagem filtram por esse campo.

**Frontend (React):**
1. `i18n.language` (gerenciado pelo `react-i18next`) é a fonte da verdade do idioma ativo.
2. Interceptor HTTP (`httpInterceptors.ts`) adiciona `Accept-Language: i18n.language` em todo request.
3. Mudança de idioma via `LanguageSettings.tsx` invalida queries do React Query para forçar refetch.
4. Componentes de exibição renderizam direto o campo já resolvido pelo backend (sem lógica adicional).
5. Forms admin (criação/edição de catálogo) usam `LocalizedTextInput` / `LocalizedTextareaInput` — abas por idioma com `pt-BR` obrigatório.

## 4. Operação Futura

### Como adicionar um novo idioma (ex: `ja-JP`)

1. **Backend `I18nConfig`:** adicionar `Locale.forLanguageTag("ja-JP")` à lista de locales suportados.
2. **Mensagens UI:** criar `messages_ja_JP.properties` em `backend/src/main/resources/messages/`.
3. **Frontend:** adicionar `ja-JP` em `frontend/src/i18n/config.ts` (`SUPPORTED_LANGUAGES`) e criar `frontend/src/i18n/locales/ja-JP/*.json` para cada namespace.
4. **Conteúdo de catálogo:** **nenhuma migração necessária**. Forms admin já aceitam novas chaves; backend já resolve qualquer tag presente no JSONB.
5. **MongoDB:** se houver busca por idioma em Title/NewsItem, criar índice composto `name_i18n.ja-JP`.
6. **Validação:** decidir se `ja-JP` é obrigatório (atualizar `@RequiredLanguages`) ou opcional (mantém só `pt-BR` obrigatório).

### Boas práticas para manter consistência

- **Sempre usar language tags BCP 47** com hífen (`pt-BR`, não `pt_BR`).
- **Fallback é responsabilidade do mapper**, não dos use cases ou controllers.
- **Use cases retornam `LocalizedString`**, nunca string resolvida — preserva todas as traduções para admin.
- **UGC nunca tem fallback** — particionamento estrito por idioma.
- **Frontend nunca envia `language` para UGC** — backend deriva do `Accept-Language`.
- **Cache HTTP de catálogo**: usar `Vary: Accept-Language` para evitar servir conteúdo errado.
- **Mudança de idioma na UI**: invalidar React Query para refetch.

### Erros comuns a evitar

- ❌ Resolver `LocalizedString` no use case → quebra DTO admin que precisa de todas as traduções.
- ❌ Aceitar `language` do frontend em UGC → permite spoofing e contaminação.
- ❌ Aplicar fallback em listagens UGC → fura o particionamento e mistura idiomas.
- ❌ Esquecer `Vary: Accept-Language` → CDN/cache serve conteúdo do idioma errado.
- ❌ Backfill sem `pt-BR` → entidades antigas ficam invisíveis para usuários em pt-BR.
- ❌ Persistir locale como `pt_BR` (underscore) em vez de `pt-BR` (hífen) → quebra match com `Locale.toLanguageTag()`.
